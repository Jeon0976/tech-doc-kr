# 6대 규칙 - 한국어 기술 문서 교정 기준

실제 문서 교정(레포 구현 노트 10편, README, 블로그)에서 독자 피드백으로 검증된 규칙이다.
기술 스택과 무관하게 적용한다 - iOS든 웹이든 백엔드든 같다.

## 1. 영어 관용구 직역 금지

영어 기술 문서에서 자연스러운 관용구를 한국어로 직역하면 아무도 안 쓰는 말이 된다.
문제는 은유 자체가 아니라 "번역된 은유"다 - 한국어 자생 표현은 살고, 직역은 죽는다.

- 목록과 대안: `calque-blacklist.md` (SSOT)
- 판별 기준: **"한국 개발자가 실제로 입에 올리는 말인가?"** 확신이 없으면 실제 한국어 블로그/포럼 용례를 검색해서 확인한다. 추측으로 판정하지 않는다.

예:
- 전: "모든 sink가 내던 세금 두 개" -> 후: "모든 sink에 반복되는 코드 두 가지"
- 전: "Sendable 판정은 움직이는 과녁이다" -> 후: "Sendable 판정은 SDK 버전에 따라 바뀐다"

## 2. 개념어는 영어 그대로, 한국어 명사를 발명하지 않는다

기술 개념에 한국어 명사를 억지로 만들면 출처가 끊기고 검색이 안 된다.
한국 개발 커뮤니티의 실제 관행도 원어 사용이다.

- 첫 등장에만 한국어 풀이를 병기하고, 이후는 영어 그대로 쓴다.
  - 예: "**structured(구조화)** = async 함수 안에서 만들어 부모에 묶이는 작업" 이후 본문은 structured.
- 실존 용어의 직역 금지: single-flight를 "단일 비행"으로 옮기면 원 출처(Go singleflight)와의 연결이 끊긴다.
- 두 번 실패하면 영어가 답이다: "고아" -> "주인 없는 작업" -> 둘 다 어색하면 unstructured Task처럼 원어로 간다.
- 약어는 첫 등장에서 풀어 쓴다: SWR -> stale-while-revalidate, TSPL -> The Swift Programming Language.

## 3. 역할 별명 대신 코드 실명

문서가 "엔진", "상자", "게이트키퍼" 같은 별명을 쓰면 독자는 그게 어느 타입인지 다시 찾아야 한다.
코드 식별자를 그대로 쓰면 문서와 코드 사이 대응 비용이 0이 된다.

- 타입/함수/변수는 실명으로: "엔진" -> `NetworkSession`, "세대 번호" -> `generation`.
- 문서 어휘 = 코드 어휘. 코드에 없는 개념어를 문서가 만들지 않는다.
- 예외: 실명이 아직 소개되지 않은 도입부에서는 일반 명사("요청을 실행하는 쪽")로 쓰고, 실명 등장 후에는 실명만 쓴다.

## 4. 형식 - 기계 생성 신호 제거

내용 이전에 형식이 먼저 읽힌다. 아래는 "사람이 타이핑하지 않았다"는 신호다.

- **비키보드 특수문자 0**: 치환표는 `special-chars.md`. 코드 인용 예외 규칙도 거기에 있다.
- **문답/면접 프레임 금지**: "예상 질문 -> 모범답안" 구조는 내용이 같아도 취준 스크립트나 AI 문답으로 읽힌다. 마무리 절은 "정리" 제목 + 평서문 불릿 + "보충 정리:" 불릿으로 쓴다.
- **압축 명사구 제목 금지**: "사실/행동 분리", "채택 밀도는 경계 배치의 결과" 같은 압축 라벨은 저자의 색인이지 독자의 언어가 아니다. 제목과 목차에서는 풀어 쓴 문장이 항상 이긴다.
- "이 문서가 답하는 질문" 같은 영어 구문 번역체 라벨 금지 -> "다루는 질문".
- 머리 블록은 자기 탐구 흐름으로: 배경 한 문장 -> 질문 불릿 -> 짧은 한 줄 결론 -> 부연 불릿. 질문과 답변이 한 덩어리 문단이면 "내 질문 + AI 답"처럼 읽힌다.

## 5. 구조 - 중복과 죽은 참조 제거

- 본문과 겹치는 절은 삭제 후보다. "오해 방지", "꼬리질문 방어", 비교표 등 선의로 넣은 절도 다른 절과 내용이 겹치면 분량만 늘린다. 지워서 나빠지는 게 없으면 지운다.
- 문서는 자기 완결적이어야 한다: 레포 밖 문서(개인 노트 제목, 시리즈 번호)를 텍스트로 참조하지 않는다. 링크할 수 있으면 링크하고, 없으면 내용을 풀어 쓴다.
- 용어를 정의 전에 쓰지 않는다. 머리 블록에서 쓰는 개념어는 그 자리에서 한 줄 풀이를 단다.

## 6. 검증 루프 - 지적 1곳은 패턴 전체의 신호

- 표현 하나가 지적되거나 탐지되면 그 자리만 고치지 않는다. **같은 표현을 대상 문서 전체(가능하면 문서군 전체)에서 전수 검색해 일괄 처리한다.** 어색한 표현은 항상 무리로 존재한다.
- 교정 후 기계 검증을 돌린다 (`scripts/verify.py`): 비키보드 문자 잔존, 피드백 마커 잔존, 상대링크 깨짐, 코드펜스 훼손.
- 불변 조건: 의미/사실/수치/고유명사 불변, 코드블록/표/링크 개수 보존, 문체(반말/존댓말) 유지. 교정은 표현만 바꾼다.
